1. OVERVIEW
-----------
This is a production-ready, enterprise-grade backend system following SOLID 
principles and Object-Oriented Programming (OOP) patterns. The architecture 
emphasizes clean code, maintainability, scalability, and testability.

Technology Stack:
  - Runtime: Node.js (ES6+ Modules)
  - Framework: Express.js
  - Database: PostgreSQL with Sequelize ORM
  - Authentication: JWT (JSON Web Tokens)
  - Password Security: bcrypt
  - File Upload: Multer
  - Process Manager: Nodemon (development)

2. ARCHITECTURE & DESIGN PATTERNS
---------------------------------

A. SOLID PRINCIPLES IMPLEMENTATION:

   1. Single Responsibility Principle (SRP)
      - Each class has ONE reason to change
      - Managers handle specific tasks (JWT, Passwords, Files)
      - Services contain business logic
      - Controllers handle HTTP I/O
      - Models define data structures
      
      Example:
        ✓ JwtManager: ONLY JWT operations
        ✓ PasswordManager: ONLY password hashing/comparison
        ✓ AuthService: ONLY authentication business logic
        ✓ AuthController: ONLY HTTP request/response handling

   2. Open/Closed Principle (OCP)
      - Open for extension, closed for modification
      - Services can be extended without modifying existing code
      - Middleware can be chained for cross-cutting concerns
      - Storage strategies can be swapped (local/S3)
      
      Example:
        ✓ UploadMiddleware can be extended to support S3
        ✓ New services can be added without modifying routes
        ✓ New validators can be added to Validator class

   3. Liskov Substitution Principle (LSP)
      - Subtypes must be substitutable for their base types
      - All middleware implementations follow Express middleware signature
      - All services implement consistent interfaces
      
      Example:
        ✓ Any storage manager can replace FileManager
        ✓ Controllers follow consistent request/response patterns

   4. Interface Segregation Principle (ISP)
      - Classes depend on specific interfaces, not general ones
      - UploadMiddleware depends on file validation, not entire file system
      - Each service has focused responsibility
      
      Example:
        ✓ AuthMiddleware only provides auth-related methods
        ✓ ContentService only handles content operations

   5. Dependency Inversion Principle (DIP)
      - Depend on abstractions, not concrete implementations
      - All dependencies are injected at initialization
      - Services don't create their own dependencies
      
      Example:
        ✓ AuthService depends on PasswordManager interface
        ✓ Routes depend on injected controllers
        ✓ Controllers depend on injected services

B. DESIGN PATTERNS:

   1. Dependency Injection (DI)
      - All dependencies injected at application startup
      - Location: ApplicationFactory.initializeDependencies()
      - Benefits: Easy testing, loose coupling, flexibility
      
   2. Factory Pattern
      - ApplicationFactory creates and configures entire app
      - Route factories create router instances with dependencies
      - Centralizes object creation logic
      
   3. Service Layer Pattern
      - Business logic separated from HTTP concerns
      - Services are reusable and testable
      - Controllers delegate to services
      
   4. Middleware Chain Pattern
      - Express middleware for cross-cutting concerns
      - Authentication, logging, error handling
      - Can be easily added/removed without changing core logic
      
   5. Repository Pattern
      - Models act as repositories for data access
      - Services use models to access data
      - Future: Easy to swap database implementations

-----------------------------

FLOW DIAGRAM:
┌─────────────────────────────────────────────────────────────┐
│ 1. LOGIN REQUEST (POST /api/auth/login)                     │
│    → Email & Password                                        │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 2. AuthService.login()                                       │
│    → Find user by email                                      │
│    → Compare password with bcrypt                            │
│    → Generate JWT token (id, email, role)                   │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 3. RESPONSE                                                  │
│    {                                                         │
│      user: {id, name, email, role},                         │
│      token: "eyJhbGc..."                                     │
│    }                                                         │
└─────────────────────────────────────────────────────────────┘

PROTECTED ROUTES:
1. AuthMiddleware.verifyToken()
   - Extracts token from Authorization header
   - Verifies token validity with JwtManager
   - Attaches user info to req.user

2. RoleMiddleware.verifyRole(...roles)
   - Checks if user.role matches required roles
   - Throws 403 error if unauthorized

ROLE HIERARCHY:
  - Principal: Full access to approval & management
  - Teacher: Upload content, view their own content
  - Student: Public API access (no authentication needed)

PASSWORD SECURITY:
  - Hashed with bcrypt (10 salt rounds)
  - Never stored in plaintext
  - Compared using bcrypt.compare()
  - JWT secrets are environment-based

6. SCHEDULING & ROTATION LOGIC (CRITICAL)
-------------------------------------------

SCHEDULING FLOW:

1. TEACHER UPLOADS CONTENT WITH SCHEDULING:
   POST /api/content/upload
   {
     "title": "Question Paper",
     "subject": "Maths",
     "start_time": "2025-04-26T10:00:00Z",
     "end_time": "2025-04-26T12:00:00Z",
     "rotation_duration": 5
   }

2. STATUS TIMELINE:
   ┌─────────────────────────────────────┐
   │ Content uploaded with scheduling    │
   │ Status: "pending"                   │
   └──────────────────┬──────────────────┘
                      │ (Principal review)
                      ▼
   ┌─────────────────────────────────────┐
   │ Principal approves content          │
   │ Status: "approved"                  │
   └──────────────────┬──────────────────┘
                      │ (Time arrives)
                      ▼
   ┌─────────────────────────────────────┐
   │ Content becomes LIVE during window  │
   │ Status: "approved" + time check     │
   └─────────────────────────────────────┘

3. ACTIVE CONTENT DETERMINATION:

   ScheduleCalculator.getActiveContent():
   
   Step 1: Filter by status = "approved"
   Step 2: Check if current time is within [start_time, end_time]
   Step 3: Calculate rotation index based on time and duration
   
   Formula:
   - earliest_start = Math.min(all start_times)
   - elapsed_ms = now - earliest_start
   - total_cycle_ms = sum(all durations) in milliseconds
   - time_in_cycle = elapsed_ms % total_cycle_ms
   
   Step 4: Find which content slot contains time_in_cycle

4. EXAMPLE CALCULATION:
   Content in rotation for "Maths":
   - Content A: duration 5 min (0-300 sec)
   - Content B: duration 3 min (300-480 sec)
   - Content C: duration 5 min (480-780 sec)
   Total cycle: 780 seconds (13 minutes)
   
   If current time at 350 seconds into cycle:
   → Between 300-480 sec → Content B is active
   
   If current time at 900 seconds into cycle:
   → 900 % 780 = 120 seconds → Content A is active

5. EDGE CASES HANDLED:
   ✓ No content available
     → Return empty response (not error)
   
   ✓ Content approved but outside time window
     → Not shown to students
   
   ✓ Content approved but no start_time/end_time
     → Treated as inactive (must have schedule)
   
   ✓ Invalid subject
     → Return empty response (not error)
   
   ✓ Rotation with single content
     → Shows same content continuously during window

8. APPROVAL WORKFLOW DESIGN
---------------------------

WORKFLOW STATES:
┌─────────────────────────────────────────────────────────────┐
│ UPLOADED STATE                                              │
│ - Content uploaded by teacher                              │
│ - Status: "pending"                                         │
│ - Waiting for principal review                             │
└──────────────────────────┬────────────────────────────────┘
                           │
         ┌─────────────────┴─────────────────┐
         │                                   │
    APPROVE                              REJECT
         │                                   │
         ▼                                   ▼
    ┌────────────────┐             ┌──────────────────┐
    │ Status:        │             │ Status:          │
    │ "approved"     │             │ "rejected"       │
    │ approved_by:   │             │ approved_by:     │
    │ {principal_id} │             │ {principal_id}   │
    │ approved_at:   │             │ rejection_reason │
    │ {timestamp}    │             │ {text}           │
    └────────────────┘             └──────────────────┘
         │
         ├─ NOT in time window → Hidden
         ├─ NO time window → Hidden
         └─ In time window → LIVE (via rotation)

APPROVAL ENDPOINTS:
  POST /api/approval/approve/:contentId
    - Principal only
    - Sets status to "approved"
    - Records principal ID and timestamp

  POST /api/approval/reject/:contentId
    - Principal only
    - Sets status to "rejected"
    - Stores rejection reason
    - Rejection reason visible to teacher

  GET /api/approval/status/:contentId
    - Any authenticated user
    - Returns full approval metadata

  GET /api/approval/stats
    - Principal only
    - Returns counts: pending, approved, rejected, total

REJECTION HANDLING:
  - Rejection reason is mandatory
  - Reason visible to uploading teacher
  - Teacher can modify and resubmit
  - Original rejection record maintained

9. PUBLIC BROADCASTING API
--------------------------

ENDPOINT: GET /api/content/live/:teacherId

DESCRIPTION:
  - Public endpoint (NO authentication)
  - Returns currently active content for a teacher
  - Respects scheduling and rotation

USAGE (By Students):
  GET /api/content/live/1        (Get all active content for teacher 1)
  GET /api/content/live/1/subject/Maths  (Get Maths content for teacher 1)

RESPONSE - Success:
  {
    "success": true,
    "statusCode": 200,
    "message": "Live content retrieved successfully",
    "data": {
      "Maths": {
        "id": 1,
        "title": "Question Paper",
        "subject": "Maths",
        "file_path": "1234567890-987654321.pdf",
        "start_time": "2025-04-26T10:00:00Z",
        "end_time": "2025-04-26T12:00:00Z"
      }
    }
  }

RESPONSE - No Content:
  {
    "success": true,
    "statusCode": 200,
    "message": "No content available",
    "data": null
  }

RESPONSE - Invalid Subject:
  {
    "success": true,
    "statusCode": 200,
    "message": "No content available",
    "data": null
  }

KEY FEATURES:
  ✓ Returns one active content per subject per teacher
  ✓ Respects teacher-defined time windows
  ✓ Automatically rotates based on duration
  ✓ No error for missing content (graceful)
  ✓ Cacheable response (optional: Redis)

10. DATABASE SCHEMA OVERVIEW
-----------------------------

USERS TABLE:
  id: INT PRIMARY KEY AUTO_INCREMENT
  name: VARCHAR(255) NOT NULL
  email: VARCHAR(255) UNIQUE NOT NULL
  password_hash: VARCHAR(255) NOT NULL
  role: ENUM('principal', 'teacher') DEFAULT 'teacher'
  created_at: DATETIME DEFAULT NOW()
  updated_at: DATETIME DEFAULT NOW()

CONTENTS TABLE:
  id: INT PRIMARY KEY AUTO_INCREMENT
  title: VARCHAR(255) NOT NULL
  description: TEXT
  subject: VARCHAR(100) NOT NULL
  file_path: VARCHAR(500) NOT NULL
  file_type: VARCHAR(50) NOT NULL
  file_size: BIGINT NOT NULL
  uploaded_by: INT NOT NULL REFERENCES users(id)
  status: ENUM('uploaded','pending','approved','rejected') DEFAULT 'pending'
  rejection_reason: TEXT
  approved_by: INT REFERENCES users(id)
  approved_at: DATETIME
  start_time: DATETIME
  end_time: DATETIME
  rotation_duration: INT DEFAULT 5 (in minutes)
  created_at: DATETIME DEFAULT NOW()
  updated_at: DATETIME DEFAULT NOW()

CONTENT_SLOTS TABLE:
  id: INT PRIMARY KEY AUTO_INCREMENT
  subject: VARCHAR(100) UNIQUE NOT NULL
  created_at: DATETIME DEFAULT NOW()

CONTENT_SCHEDULES TABLE:
  id: INT PRIMARY KEY AUTO_INCREMENT
  content_id: INT NOT NULL REFERENCES contents(id)
  slot_id: INT NOT NULL REFERENCES content_slots(id)
  rotation_order: INT DEFAULT 0
  duration: INT DEFAULT 5 (in minutes)
  created_at: DATETIME DEFAULT NOW()

INDEXES:
  - contents(uploaded_by, status)
  - contents(subject, status, start_time)
  - content_schedules(slot_id, rotation_order)

11. ERROR HANDLING & VALIDATION
-------------------------------

ERROR HIERARCHY:
  AppError (Custom)
    ├── 400 Bad Request (Validation errors)
    ├── 401 Unauthorized (Auth failures)
    ├── 403 Forbidden (Permission denied)
    ├── 404 Not Found (Resource not found)
    ├── 409 Conflict (Resource exists)
    └── 500 Internal Server Error

ERROR RESPONSE FORMAT:
  {
    "success": false,
    "statusCode": 400,
    "message": "Validation error description",
    "errors": null or {detailed_errors}
  }

VALIDATION RULES:
  Email:
    ✓ Valid email format required
    ✓ Unique in database

  Password:
    ✓ Minimum 6 characters
    ✓ Hashed with bcrypt before storage

  Subject:
    ✓ Predefined list: [Maths, Science, English, History, Geography]
    ✓ Case-sensitive

  Role:
    ✓ Must be "principal" or "teacher"

  File:
    ✓ Must be JPG, PNG, or GIF
    ✓ Max 10MB
    ✓ Required field

  Scheduling:
    ✓ start_time < end_time
    ✓ start_time cannot be in past
    ✓ rotation_duration >= 1 minute

VALIDATION IMPLEMENTATION:
  - Validator class with static methods
  - Validates at controller/service boundary
  - Centralized, reusable validation logic
  - Custom error messages
