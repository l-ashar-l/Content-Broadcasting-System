2. ARCHITECTURE & DESIGN PATTERNS
---------------------------------

A. SOLID PRINCIPLES IMPLEMENTATION:

   1. Single Responsibility Principle (SRP)
      Each class has ONE reason to change.
      
      Examples:
        ✓ JwtManager: ONLY JWT operations
        ✓ PasswordManager: ONLY password hashing/comparison
        ✓ RedisManager: ONLY cache operations
        ✓ AuthService: ONLY authentication business logic
        ✓ AnalyticsService: ONLY analytics queries and aggregation
        ✓ AuthController: ONLY HTTP request/response handling
        ✓ AnalyticsController: ONLY analytics endpoint handling

   2. Open/Closed Principle (OCP)
      Open for extension, closed for modification.
      
      Examples:
        ✓ Services can be extended without modifying existing code
        ✓ Middleware can be chained without changing core logic
        ✓ Storage strategies swappable (local/S3)
        ✓ New analytics endpoints added without modifying old ones
        ✓ Rate limiters configurable per endpoint

   3. Liskov Substitution Principle (LSP)
      Subtypes substitutable for their base types.
      
      Examples:
        ✓ All middleware follows Express middleware signature
        ✓ All services implement consistent patterns
        ✓ Any storage manager can replace FileManager
        ✓ Cache layer is swappable (Redis or no-op)

   4. Interface Segregation Principle (ISP)
      Classes depend on specific interfaces, not general ones.
      
      Examples:
        ✓ AuthMiddleware only provides auth methods
        ✓ AnalyticsService only handles analytics operations
        ✓ RedisManager only exposes cache operations
        ✓ Controllers have focused responsibilities

   5. Dependency Inversion Principle (DIP)
      Depend on abstractions, not concrete implementations.
      
      Examples:
        ✓ All dependencies injected in ApplicationFactory
        ✓ Controllers don't instantiate services
        ✓ Services don't create their own dependencies
        ✓ RedisManager injected into BroadcastController
        ✓ Rate limiters injected into route factories

B. DESIGN PATTERNS:

   1. Dependency Injection (DI)
      All dependencies injected at application startup.
      Location: ApplicationFactory.initializeDependencies()
      
      Benefits: Easy testing, loose coupling, flexibility
      
      Injected Components:
        - Managers (JWT, Password, S3, Redis)
        - Services (Auth, Content, Approval, Analytics, Rotation)
        - Controllers (Auth, Content, Approval, Analytics, Broadcast)
        - Middleware (Auth, Upload, Rate Limiting)

   2. Factory Pattern
      ApplicationFactory creates and configures entire app.
      Route factories create router instances with dependencies.
      Centralizes object creation logic.
      
      Examples:
        ✓ ApplicationFactory.initializeApp()
        ✓ createAuthRoutes(controller, middleware)
        ✓ createAnalyticsRoutes(controller, middleware, rateLimiter)

   3. Service Layer Pattern
      Business logic separated from HTTP concerns.
      Services are reusable and testable.
      Controllers delegate to services.
      
      Services:
        - AuthService (login, register, token refresh)
        - ContentService (CRUD, scheduling)
        - ApprovalService (approval workflows)
        - AnalyticsService (usage tracking, statistics)
        - RotationService (content scheduling logic)

   4. Middleware Chain Pattern
      Express middleware for cross-cutting concerns.
      Authentication, logging, error handling, rate limiting.
      Can be easily added/removed without changing core logic.
      
      Middleware:
        - RequestLogger (logs all requests)
        - CORS (cross-origin configuration)
        - ErrorHandler (centralized error handling)
        - AuthMiddleware (JWT verification, role checking)
        - RateLimiters (tiered rate limiting)
        - S3UploadMiddleware (file validation)

   5. Singleton Pattern
      RedisManager maintains single Redis connection.
      Prevents connection pooling issues.
      Graceful fallback if Redis unavailable.

   6. Observer/Event Pattern
      Redis client events (connect, error, disconnect).
      Ensures proper connection state management.

4. FEATURE IMPLEMENTATIONS
--------------------------

A. REDIS CACHING (✨ NEW)
   
   Location: src/utils/RedisManager.js
   
   Purpose:
     - Improve response time for frequently accessed data
     - Reduce database load
     - Support high-traffic scenarios
   
   Implementation:
     - Singleton RedisManager maintains single connection
     - BroadcastController caches live content endpoints
     - 5-minute TTL for cached live content
     - Graceful degradation if Redis unavailable
   
   Cache Keys:
     live_content_{teacherId}
     live_content_{teacherId}_{subject}
   
   Configuration:
     REDIS_HOST=redis (Docker service name)
     REDIS_PORT=6379
   
   Usage Pattern:
     1. Check cache for key
     2. If hit, return cached data
     3. If miss, fetch from DB
     4. Cache result with TTL
     5. Return to client
   
   Endpoints Cached:
     GET /api/content/live/:teacherId
     GET /api/content/live/:teacherId/subject/:subject

B. MULTI-TIER RATE LIMITING (✨ NEW)
   
   Location: src/middlewares/rate-limit.middleware.js
   
   Purpose:
     - Protect APIs from abuse and brute force attacks
     - Ensure fair usage across users
     - Prevent DDoS and resource exhaustion
   
   Implementation:
     Four configurable limiters based on endpoint type:
     
     1. publicApiLimiter
        - Limit: 10 requests per 15 minutes
        - Applied to: All /api/* endpoints
        - Skips: Authenticated requests
     
     2. authLimiter
        - Limit: 5 requests per 15 minutes
        - Applied to: /api/auth/login, /api/auth/register
        - Purpose: Prevent brute force attacks
     
     3. uploadLimiter
        - Limit: 20 requests per hour
        - Applied to: /api/content/upload
        - Rate by: User ID (if authenticated) or IP
     
     4. analyticsLimiter
        - Limit: 30 requests per hour
        - Applied to: /api/analytics/*
        - Rate by: IP address
   
   Response Headers:
     RateLimit-Limit: total allowed requests
     RateLimit-Remaining: requests left in window
     RateLimit-Reset: Unix timestamp when limit resets
   
   Error Response (429 Too Many Requests):
     {
       "success": false,
       "statusCode": 429,
       "message": "Too many requests, please try again later"
     }

C. ANALYTICS SYSTEM (✨ NEW)
   
   Location: 
     - src/models/ContentUsage.js (data model)
     - src/services/AnalyticsService.js (business logic)
     - src/controllers/AnalyticsController.js (HTTP handlers)
     - src/routes/analytics.routes.js (API endpoints)
   
   Purpose:
     - Track content usage and engagement metrics
     - Provide insights into subject popularity
     - Monitor user behavior patterns
   
   Data Model (content_usages table):
     - id (PK): Unique usage record ID
     - content_id (FK): Reference to content
     - action (ENUM): view | download | share
     - user_id (FK, nullable): Reference to user
     - ip_address: Client IP address
     - created_at: Timestamp of action
   
   Endpoints:
     
     1. POST /api/analytics/usage (PUBLIC)
        Purpose: Record a usage event
        Rate Limit: 30 req/hour
        Auth: None required
        Validation: Content must exist
        
        Request:
          { "contentId": 1, "action": "view" }
        
        Response:
          { "success": true, "data": { usage record } }
     
     2. GET /api/analytics/subjects/most-active (PROTECTED)
        Purpose: Get top N subjects by usage count
        Rate Limit: 30 req/hour
        Auth: JWT required
        Query Params: limit (default: 5)
        
        Returns: List of subjects with view/download/share counts
     
     3. GET /api/analytics/subjects (PROTECTED)
        Purpose: Get analytics for all subjects
        Rate Limit: 30 req/hour
        Auth: JWT required
        
        Returns: Each subject's: total content, total usage, views, 
                 downloads, active users, last access timestamp
     
     4. GET /api/analytics/content/:id/usage (PROTECTED)
        Purpose: Get detailed stats for specific content
        Rate Limit: 30 req/hour
        Auth: JWT required
        
        Returns: Content info + statistics + recent usage (10 most recent)
     
     5. GET /api/analytics/trend (PROTECTED)
        Purpose: Get usage trends over time
        Rate Limit: 30 req/hour
        Auth: JWT required
        Query Params: contentId, subject, days (default: 7)
        
        Returns: Daily breakdown of views, downloads, unique users
   
   Query Strategy:
     - Uses raw SQL for complex aggregations (better performance)
     - Handles NULL joins gracefully (LEFT JOINs)
     - Parameterized queries to prevent SQL injection
     - Proper table name quoting ("contents", "content_usages")

PROTECTED ROUTES:
1. AuthMiddleware.verifyToken()
   - Extracts token from Authorization header
   - Verifies token validity with JwtManager
   - Attaches user info to req.user

2. RoleMiddleware.verifyRole(...roles)
   - Checks if user.role matches required roles
   - Throws 403 error if unauthorized

ROLE HIERARCHY:
  - Principal: Full access to approval & management
  - Teacher: Upload content, view their own content
  - Student: Public API access (no authentication needed)

  - Never stored in plaintext
  - Compared using bcrypt.compare()
  - JWT secrets are environment-based

6. SCHEDULING & ROTATION LOGIC (CRITICAL)
-------------------------------------------

SCHEDULING FLOW:

1. TEACHER UPLOADS CONTENT WITH SCHEDULING:
   POST /api/content/upload
   {
     "title": "Question Paper",
     "subject": "Maths",
     "start_time": "2025-04-26T10:00:00Z",
     "end_time": "2025-04-26T12:00:00Z",
     "rotation_duration": 5
   }

2. STATUS TIMELINE:
   ┌─────────────────────────────────────┐
   │ Content uploaded with scheduling    │
   │ Status: "pending"                   │
   └──────────────────┬──────────────────┘
                      │ (Principal review)
                      ▼
   ┌─────────────────────────────────────┐
   │ Principal approves content          │
   │ Status: "approved"                  │
   └──────────────────┬──────────────────┘
                      │ (Time arrives)
                      ▼
   ┌─────────────────────────────────────┐
   │ Content becomes LIVE during window  │
   │ Status: "approved" + time check     │
   └─────────────────────────────────────┘

3. ACTIVE CONTENT DETERMINATION:

   ScheduleCalculator.getActiveContent():
   
   Step 1: Filter by status = "approved"
   Step 2: Check if current time is within [start_time, end_time]
   Step 3: Calculate rotation index based on time and duration
   
   Formula:
   - earliest_start = Math.min(all start_times)
   - elapsed_ms = now - earliest_start
   - total_cycle_ms = sum(all durations) in milliseconds
   - time_in_cycle = elapsed_ms % total_cycle_ms
   
   Step 4: Find which content slot contains time_in_cycle

4. EXAMPLE CALCULATION:
   Content in rotation for "Maths":
   - Content A: duration 5 min (0-300 sec)
   - Content B: duration 3 min (300-480 sec)
   - Content C: duration 5 min (480-780 sec)
   Total cycle: 780 seconds (13 minutes)
   
   If current time at 350 seconds into cycle:
   → Between 300-480 sec → Content B is active
   
   If current time at 900 seconds into cycle:
   → 900 % 780 = 120 seconds → Content A is active

5. EDGE CASES HANDLED:
   ✓ No content available
     → Return empty response (not error)
   
   ✓ Content approved but outside time window
     → Not shown to students
   
   ✓ Content approved but no start_time/end_time
     → Treated as inactive (must have schedule)
   
   ✓ Invalid subject
     → Return empty response (not error)
   
   ✓ Rotation with single content
     → Shows same content continuously during window

  POST /api/approval/approve/:contentId
    - Principal only
    - Sets status to "approved"
    - Records principal ID and timestamp

  POST /api/approval/reject/:contentId
    - Principal only
    - Sets status to "rejected"
    - Stores rejection reason
    - Rejection reason visible to teacher

  GET /api/approval/status/:contentId
    - Any authenticated user
    - Returns full approval metadata

  GET /api/approval/stats
    - Principal only
    - Returns counts: pending, approved, rejected, total

REJECTION HANDLING:
  - Rejection reason is mandatory
  - Reason visible to uploading teacher
  - Teacher can modify and resubmit
  - Original rejection record maintained

10. DATABASE SCHEMA OVERVIEW
-----------------------------

USERS TABLE:
  id: INT PRIMARY KEY AUTO_INCREMENT
  name: VARCHAR(255) NOT NULL
  email: VARCHAR(255) UNIQUE NOT NULL
  password_hash: VARCHAR(255) NOT NULL
  role: ENUM('principal', 'teacher') DEFAULT 'teacher'
  created_at: DATETIME DEFAULT NOW()
  updated_at: DATETIME DEFAULT NOW()

CONTENTS TABLE:
  id: INT PRIMARY KEY AUTO_INCREMENT
  title: VARCHAR(255) NOT NULL
  description: TEXT
  subject: VARCHAR(100) NOT NULL
  file_path: VARCHAR(500) NOT NULL
  file_type: VARCHAR(50) NOT NULL
  file_size: BIGINT NOT NULL
  uploaded_by: INT NOT NULL REFERENCES users(id)
  status: ENUM('uploaded','pending','approved','rejected') DEFAULT 'pending'
  rejection_reason: TEXT
  approved_by: INT REFERENCES users(id)
  approved_at: DATETIME
  start_time: DATETIME
  end_time: DATETIME
  rotation_duration: INT DEFAULT 5 (in minutes)
  created_at: DATETIME DEFAULT NOW()
  updated_at: DATETIME DEFAULT NOW()

CONTENT_SLOTS TABLE:
  id: INT PRIMARY KEY AUTO_INCREMENT
  subject: VARCHAR(100) UNIQUE NOT NULL
  created_at: DATETIME DEFAULT NOW()

CONTENT_SCHEDULES TABLE:
  id: INT PRIMARY KEY AUTO_INCREMENT
  content_id: INT NOT NULL REFERENCES contents(id)
  slot_id: INT NOT NULL REFERENCES content_slots(id)
  rotation_order: INT DEFAULT 0
  duration: INT DEFAULT 5 (in minutes)
  created_at: DATETIME DEFAULT NOW()

INDEXES:
  - contents(uploaded_by, status)
  - contents(subject, status, start_time)
  - content_schedules(slot_id, rotation_order)
